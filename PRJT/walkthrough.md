# 작업 완료 보고서 (Walkthrough)

`tax_business_logic.md`에 명시된 비즈니스 룰을 바탕으로 백엔드 세금 계산 로직을 구현하고, DB 스키마 및 프론트엔드 연동 작업을 완료하였습니다.

## 주요 변경 사항

### 1. 세금 계산 모듈 추가 (`backend/logic/tax_calc.py`)
- **[Rule A] 부가가치세 (VAT) 계산**: 
  - 연 매출 1억 400만 원 기준 '간이과세자'(0원)와 '일반과세자' 분기 처리 및 `(매출액 * 10%) - (적격증빙 매입액 * 10%)` 공식 적용.
- **[Rule B] 필요경비 분석 및 세복이 경고 시스템**:
  - 적격증빙(`세금계산서`, `계산서`, `신용카드`, `현금영수증`)에 대해서만 100% 필요경비 인정.
  - 비적격증빙(`간이영수증`, `증빙없음`) 중 **30,000원 초과 건**에 대해 세복이 맞춤형 경고 메시지 배열(`seboki_warnings`) 생성.
- **[Rule C] 노란우산공제 한도 적용**:
  - 사업소득금액(매출 - 필요경비) 구간별로 500만 원 / 300만 원 / 200만 원 공제 한도 차등 적용.
- **과세표준 산출**: 사업소득금액에서 한도 적용된 노란우산공제액을 차감하여 최종 `taxable_income` 도출.

### 2. 백엔드 연동 (`backend/main.py`)
- **DB 스키마 마이그레이션**: `transactions` 테이블에 `proof_type` 컬럼을 추가하고, 기존 DB에서도 호환되도록 자동 마이그레이션(`ALTER TABLE`)을 처리했습니다.
- **모의 데이터 고도화**: `/api/connect-accounts` 시 모의 거래 데이터에 적절한 `proof_type`을 명시하고, 3만 원 초과 간이영수증 건을 포함시켜 세복이 경고가 자연스럽게 동작하도록 설정했습니다.
- **대시보드 API 개편**: `/api/dashboard`에서 실제 장부 내역을 쿼리하여 매출/매입 데이터를 생성하고, `calculate_estimated_tax` 함수를 활용해 실시간으로 세금과 피드백 경고를 산출하여 반환하도록 변경했습니다.
- **영수증 스캔 및 벌크 API 업데이트**: 영수증 AI 분석 및 DB 일괄 저장 시 `proof_type`을 올바르게 바인딩하도록 수정하였습니다.

### 3. 프론트엔드 연동 (`html/index.html` 및 `js/app.js`)
- **세복이 말풍선 경고 노출**: 대시보드의 세복이 말풍선 영역에서 백엔드로부터 받아온 `seboki_warnings`의 경고 리스트를 실시간으로 출력하여 사용자에게 비적격증빙의 비용처리 부인 위험을 즉각 인지시킵니다.
- **영수증 AI 분석 모달 증빙 종류 수정**: AI 분석 결과 팝업 내에 세무 분류뿐만 아니라 **증빙 종류**(`세금계산서`, `계산서`, `신용카드`, `현금영수증`, `간이영수증`, `증빙없음`)를 사용자가 직접 확인하고 수정할 수 있도록 선택창(Select)을 배치하였습니다.
- **저장 데이터 바인딩**: 장부 저장 시 수정된 `proof_type` 정보가 백엔드에 안전하게 보관되도록 변경했습니다.

## 테스트 및 검증 결과
- **FastAPI 구동 확인**: FastAPI 서버가 구문 에러 없이 정상 작동하며, `/api/dashboard` 호출 시 새로 규정된 비즈니스 룰 기반 예상 세금 및 세복이 경고 리스트(`seboki_warnings`)가 올바르게 반환되는 것을 검증하였습니다.

## 추가 개선 사항 (영수증 OCR 추출 고도화)
- **가맹점명(상호명) 추출 로직 개선**:
  - `index.html`, `js/app.js`, `backend/main.py`에 `"농협", "하나로마트", "축협", "수협", "신협", "이마트24"` 등 자주 쓰이는 한국 가맹점 브랜드를 검색 키워드(`store_keywords`)에 추가했습니다.
  - OCR 인식 시 로고 기호가 영문 노이즈(`FAD` 등)로 잘못 추출되는 현상을 방지하기 위해, 우선순위 3의 후보 추출 조건에 한글 포함 필수 조건(`/[가-힣]/` 패턴)을 지정하였습니다.
  - 전체 OCR 텍스트 내에 홈페이지 도메인 등으로 `"nonghyup"` 또는 `"농협"` 키워드가 존재하지만 상호명이 비어있거나 `FAD` 등으로 오인된 경우, 이를 자동으로 `"농협"`으로 변환/폴백해주는 보정 필터를 탑재했습니다.
  - **배포 환경(OpenAI Vision API) 대응**: 배포된 서버에서 동작하는 OpenAI Vision API의 시스템 프롬프트에 농협 소 얼굴 로고가 `FAD` 등으로 misread 될 수 있음을 명증하는 지침을 추가하였고, API 반환 결과의 `summary`에서도 `FAD`가 존재할 시 `농협`으로 안전하게 치환하는 후처리 보정을 적용하여 배포(OpenAI) 환경에서의 가맹점 인식 실패를 근본적으로 해결했습니다.
- **결제 금액 추출 필터 정밀화**:
  - 품목 바코드 번호 등의 노이즈가 총액으로 인식되는 일을 막고자, 바코드에 자주 쓰이는 `*` 문자가 포함된 라인은 추출 대상에서 제외했습니다.
  - 카드번호, 사업자 가맹점 번호, 지역 번호 등 leading zero로 시작하는 임의의 수치(`00000074900` 등)가 금액으로 잘못 판정되지 않도록 앞자리가 `0`인 패턴을 필터링하였습니다.
  - 영수증 전체에서 동일한 가격 수치가 중복해서 발견되는 빈도(Frequency)를 측정해, 가장 많이 반복 등장한 최종 금액(예: `8,560`원)에 가중치를 가산하여 노이즈 대비 가장 신뢰도 높은 합계액을 추출하도록 구현했습니다.

## 추가 개선 사항 (영수증 결과 모달의 세무 분류 수정 오동작 해결)
- **React 상태 업데이트 경쟁 상태(Race Condition) 해결**:
  - 모달 내에서 세무 분류 수정 dropdown을 변경할 때 `handleResultChange(idx, 'category', ...)`와 `handleResultChange(idx, 'category_kr', ...)`를 연속해서 호출하고 있었습니다.
  - React의 비동기 상태 업데이트 특성상, 두 번째 호출에서 이전의 오래된 `receiptResults` 상태를 참조하여 첫 번째 호출에서 변경한 `category` 값이 덮어씌워지는 경쟁 상태가 발생했습니다.
  - 이를 해결하기 위해 `handleResultChange` 내부에서 **함수형 업데이트(`setReceiptResults(prev => ...)`)** 방식을 적용하여 항상 가장 최신의 상태를 기반으로 변경이 누적되도록 개선하였으며, 한 번의 호출로 객체 다중 필드 업데이트가 가능하도록 확장하고 dropdown `onChange` 이벤트 시 단일 객체 전달 방식으로 리팩토링했습니다. 이로써 사용자가 "사업용"으로 수정한 세분류 정보가 유실되지 않고 장부에 안전하게 저장됩니다.

## 추가 개선 사항 (대시보드 예상 세금과 세금 신고 산출 세액 불일치 문제 해결)
- **세액 계산 불일치 원인 분석**:
  - **대시보드 세금 예상 상세 보고서(첫 번째 사진)**: 실제 세법을 기반으로 간이과세 기준(매출 1.04억 미만 시 부가세 0원), 종합소득세 누진세율 및 **노란우산공제(300만 원)**를 적용해 예상 납부 세액을 `0원`으로 정상 산출하고 있었습니다.
  - **세금 신고 2단계의 실시간 세액 산출 내역(두 번째 사진)**: 노란우산공제 차감이나 간이과세 면제 여부를 고려하지 않고 단순 수식 `(소득금액) * 10%` 방식을 사용해 `1,214,000 * 10% = 121,400원`을 산출하면서 불일치가 발생했습니다.
- **통합 정합성 수정**:
  - `index.html` 및 `js/app.js` 내 `calculateFilingStats()` 함수의 최종 산출 세액(`calculatedTax`)을 대시보드 및 세무 룰이 온전히 적용된 실제 예상 세액(`dashboard.expected_tax`)을 직접 연동하도록 변경했습니다. 이로써 대시보드의 실 예상 세금과 전자 신고 단계의 산출 세액이 완벽히 일치하도록 조치했습니다.
  - **부가세 세율 10% 일괄 적용**: 기존의 연 매출 1억 400만 원 미만일 때 부가세를 `0원`으로 처리하던 코드(`backend/tax_calc.py`)를 수정하여, 간이과세자 여부와 관계없이 부가세 세율 10%를 일괄 적용해 `(매출액 * 10%) - (적격증빙 매입액 * 10%)`로 세금이 정상 산출되도록 일치시켰습니다. 이로써 신고 화면과 대시보드 모두 부가세 10% 룰에 따라 `129,300`원(총 지출 2,656,000원 중 적격증빙 인정액 2,577,000원 제외)의 일관된 예상 세액 결과를 갖게 됩니다.
